## 二、系统架构设计

### 2.1 整体架构模式

系统采用 **C-S架构（客户端-服务端架构）**，基于**多进程+多线程**的分布式设计：

```
┌─────────────────────────────────────────────────────────┐
│                    C++ 服务端 (apiserver)                │
│  ┌──────────────┐  ┌──────────────┐  ┌──────────────┐  │
│  │ 行情引擎线程 │  │ 交易引擎线程 │  │ 数据记录线程  │  │
│  │ (MD Engine)  │  │ (TD Engine)  │  │ (Recorder)   │  │
│  └──────────────┘  └──────────────┘  └──────────────┘  │
│         │                  │                  │         │
│         └──────────────────┼──────────────────┘         │
│                            │                            │
│              ┌─────────────▼─────────────┐             │
│              │   消息队列中心 (MsgCenter) │             │
│              │  PUB/SUB + PUSH/PULL      │             │
│              └─────────────┬─────────────┘             │
└────────────────────────────┼────────────────────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
┌───────▼──────┐    ┌────────▼────────┐  ┌───────▼──────┐
│ 策略进程     │    │  GUI界面进程     │  │ 数据记录进程 │
│ (strategy.py)│    │  (gui.py)        │  │ (recorder.py)│
│              │    │                  │  │              │
│ StrategyEngine│   │  MainWindow      │  │ RecorderEngine│
│              │    │  (PyQt5)         │  │              │
└──────────────┘    └──────────────────┘  └──────────────┘
```

### 2.3 通信架构

#### 2.3.1 消息队列通信（默认模式）

**协议**: nanomsg (支持ZMQ)

**通信模式**:
1. **PUB/SUB (发布/订阅)**
   - 服务端 → 客户端（行情、订单状态、持仓等）
   - URL: `tcp://localhost:55555` (SERVERPUB_URL)

2. **PUSH/PULL (推拉)**
   - 客户端 → 服务端（下单、查询请求）
   - URL: `tcp://localhost:55557` (SERVERPULL_URL)

3. **SUB (订阅)**
   - 服务端内部订阅
   - URL: `tcp://localhost:55556` (SERVERSUB_URL)

**消息格式**: `目的地|源地址|消息类型|消息内容`

**延迟性能**:
- 消息队列模式: 30-100微秒延迟
- 吞吐量: 约10k/s（对期货交易足够）

#### 2.3.2 共享内存通信（高性能模式）

- 延迟: **10微秒级别**
- 吞吐量: **10万/秒**
- 适用于: 股票L2逐笔数据等高吞吐场景

### 2.4 数据存储架构

**支持的数据源**:
1. **MongoDB** - 主要数据库（支持批量读写，性能提升2-10倍）
2. **SQLite/MySQL/PostgreSQL** - 关系型数据库支持
3. **CSV文件** - 简单数据存储
4. **内存模式** - 回测时的高性能模式

**数据记录**:
- Tick数据实时记录
- Bar数据聚合记录
- 交易记录完整保存
- 支持数据导入导出

### 4.3 编写策略

#### 4.3.1 策略模板

在 `mystrategy/` 目录下创建策略文件：

```python
from source.strategy.strategy_base import StrategyBase
from source.common.constant import Interval

class MyStrategy(StrategyBase):
    """我的策略"""
    
    # 策略参数（可在GUI中修改）
    parameters = ["api", "account", "fast_window", "slow_window"]
    api = "CTP.TD"
    account = "123456"
    fast_window = 5
    slow_window = 20
    
    # 策略变量（会持久化保存）
    variables = ["fast_ma", "slow_ma"]
    fast_ma = 0
    slow_ma = 0
    
    def on_init(self):
        """策略初始化"""
        self.write_log("策略初始化")
        # 加载历史数据
        self.load_bar(10, Interval.MINUTE)
    
    def on_start(self):
        """策略启动"""
        self.write_log("策略启动")
    
    def on_bar(self, bar):
        """K线更新"""
        # 计算指标
        # 交易逻辑
        if self.fast_ma > self.slow_ma:
            if self.pos == 0:
                self.buy_open(bar.close_price, 1)
        elif self.fast_ma < self.slow_ma:
            if self.pos > 0:
                self.sell_close(bar.close_price, self.pos)
    
    def on_tick(self, tick):
        """Tick更新（可选）"""
        pass
    
    def on_order(self, order):
        """订单状态更新"""
        self.write_log(f"订单状态: {order.status}")
    
    def on_trade(self, trade):
        """成交回报"""
        self.write_log(f"成交: {trade.price}, {trade.volume}")
```

#### 4.3.2 策略API说明

**下单函数**:
- `buy_open(price, size, type='lmt')` - 买入开仓
- `sell_close(price, size, type='lmt')` - 卖出平仓
- `sell_open(price, size, type='lmt')` - 卖出开仓
- `buy_close(price, size, type='lmt')` - 买入平仓

**订单类型**:
- `'lmt'` - 限价单
- `'mkt'` - 市价单
- `'fak'` - FAK（立即成交剩余撤销）
- `'fok'` - FOK（全部成交或全部撤销）

**数据加载**:
- `load_bar(days, interval, callback, datasource='DataBase')` - 加载K线数据
- `load_tick(days, datasource='DataBase')` - 加载Tick数据

**查询函数**:
- `get_tick(full_symbol)` - 获取最新Tick
- `get_position_holding(account, full_symbol)` - 获取持仓
- `get_account(accountid)` - 获取账户信息

#### 4.4.2 代码回测

```python
from source.engine.backtest_engine import BacktestingEngine
from mystrategy.my_strategy import MyStrategy
from datetime import datetime

# 创建回测引擎
engine = BacktestingEngine()

# 设置回测参数
engine.set_parameters(
    full_symbol="SHFE F RB 1905",
    interval="1m",
    start=datetime(2019, 1, 1),
    rate=0.0001,  # 手续费率
    slippage=1,  # 滑点（跳）
    size=10,  # 合约乘数
    pricetick=1,  # 最小价格变动
    capital=1000000,  # 初始资金
    end=datetime(2019, 12, 31),
    mode=BacktestingMode.BAR
)

# 添加策略
engine.add_strategy(MyStrategy, {
    "fast_window": 5,
    "slow_window": 20
})

# 加载数据
engine.load_data()

# 运行回测
engine.run_backtesting()

# 计算结果
engine.calculate_result()

# 计算统计指标
statistics = engine.calculate_statistics()
print(f"总收益率: {statistics['total_return']:.2f}%")
print(f"夏普率: {statistics['sharpe_ratio']:.2f}")
print(f"最大回撤: {statistics['max_ddpercent']:.2f}%")
```

### 5.2 面试官可能提出的问题及解答

#### Q1: 为什么选择C++和Python混合架构？

**A**: 
- **性能需求**：行情和交易接口需要处理高频数据，C++能提供微秒级延迟
- **开发效率**：策略开发需要快速迭代，Python语法简洁，生态丰富
- **资源利用**：C++服务端可以充分利用CPU和内存，Python策略进程可以独立扩展
- **风险隔离**：Python策略崩溃不会影响C++服务端稳定性

#### Q2: 消息队列为什么选择nanomsg而不是其他方案？

**A**:
- **轻量级**：nanomsg比ZMQ更轻量，依赖少
- **性能**：延迟在30-100微秒，满足期货交易需求
- **简单**：API简洁，易于集成
- **灵活性**：同时支持共享内存模式，可扩展到10微秒级延迟

#### Q3: 如何保证实盘交易的安全性？

**A**:
1. **风险控制模块**：
   - 流控（每秒订单数限制）
   - 总量控制（单笔订单大小、总持仓限制）
   - 资金控制（单笔资金、总资金限制）

2. **进程隔离**：
   - 策略进程崩溃不影响交易接口
   - 服务端自动重连机制

3. **订单管理**：
   - 完整的订单状态跟踪
   - 持仓和账户实时同步
   - 异常订单自动拒绝

4. **日志记录**：
   - 所有交易操作完整记录
   - 便于问题追溯和审计

#### Q4: 回测和实盘的差异如何处理？

**A**:
1. **统一接口**：策略基类StrategyBase在回测和实盘中完全一致

3. **数据源抽象**：支持从数据库或内存加载数据
4. **撮合逻辑**：回测中模拟真实撮合（考虑滑点、手续费）
5. **实盘验证**：Tick级回测与实盘成交时间和价位一致

#### Q5: 如何处理高并发场景？

**A**:
1. **多进程架构**：每个策略独立进程，天然并发

3. **共享内存模式**：PRO版支持共享内存，10万/s吞吐量
